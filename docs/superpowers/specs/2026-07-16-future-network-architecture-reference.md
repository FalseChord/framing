# 參考文件：對外／遠端存取架構（未實作，留供後續階段參考）

本文件保存的是**簡化為 Windows 本機測試前**討論過的完整架構方向，涵蓋雲端方案的評估過程與被否決的原因，以及「之後要開放遠端存取」時已經談過的取捨。目前的正式設計請見 `2026-07-16-letter-templates-design.md`；本文件純粹是決策脈絡與未來參考，不是目前要實作的規格。

## 為什麼不用雲端服務

專案一開始評估過 Firebase、Vercel + Postgres、Supabase 等雲端方案，過程中發現的關鍵問題：

- **Firestore（文件型資料庫）**：使用者評估後認為遷移風險高（不易匯出成關聯式資料，未來個案/心理師/預約的關聯資料結構不適合），予以否決。
- **Firebase Cloud Functions（無伺服器）**：沒有可靠的本機硬碟可持久化資料，與「想用 SQLite/JSON 檔案」的想法衝突。
- **Vercel Hobby（免費層）**：服務條款限「非商業用途」，諮商所屬於商業機構，嚴格認定上不符合；升級 Pro 方案約每月 $20 美金起。
- **Vercel Edge Functions**：使用者過去在 GCP Cloud Run 有過冷啟動十幾秒的不良經驗，因此傾向避開 edge runtime，改用一般 Node.js Serverless Functions（重新評估後，Vercel 的 Node Function 冷啟動與 Cloud Run 容器冷啟動是不同機制，通常快得多，但使用者的疑慮仍然合理，值得保留紀錄）。
- **DPA／資料保護合約**：免費方案普遍沒有正式的資料處理協議（DPA），這類合約多半是付費方案才有；Supabase 一度被認為可能符合需求（其標準 DPA 據稱各方案皆可簽署，但條款會變動，需自行至官網確認最新狀態，不應以此對話內容為準）。
- **最終關鍵問題：台灣個資法跨境傳輸疑慮**。無論選哪個雲端商，資料離開機構等於「委外處理／可能構成跨境傳輸」，而機構現有的個案同意書並未涵蓋雲端／第三方處理，且歷史資料在現有同意基礎上可能無法上雲。GCP 台灣機房（asia-east1，彰化）雖然資料實體位置在台灣，但 Google Cloud 是美國公司，是否因此就不算跨境傳輸，屬於需要法律專業判斷的灰色地帶，非技術選型可以解決。

因法務判斷結果指向自架，本專案改為**自架在機構自有硬體**，完全避開委外處理的問題（見主設計文件）。

## 對外／遠端存取的取捨（尚未實作）

若之後（例如系統穩定後，或需要讓行政人員在家/從自己電腦連線）需要對外開放存取，已經討論過三種方案：

### 方案 A：直接開 port + 安全硬化（目前傾向的方向）
- 反向代理（建議 Caddy，自動處理 HTTPS 憑證）
- 需要網域名稱指向公網 IP（買一個便宜網域，或先用免費 DDNS 服務如 DuckDNS）
- 防火牆只開 443；SSH（若需要）走非預設 port + 金鑰登入 + fail2ban
- 系統自動安全更新
- **優點**：使用者端零安裝，開瀏覽器打網址即可用
- **缺點**：伺服器會被公網掃描到，維運責任（修補、憑證、防暴力破解）在自己身上，攻擊面比方案 B 大

### 方案 B：私人 VPN（Tailscale）— 已否決
- 伺服器與每台要連線的裝置都安裝 Tailscale，只有裝置在同一個 tailnet 才能連到伺服器，伺服器完全不對公網開放任何 port
- **優點**：攻擊面幾乎為零，資料不經任何第三方中繼查看內容
- **缺點/否決原因**：需要每位使用者的裝置額外安裝一次軟體，使用者判斷這個「安裝一次」的 effort 不值得換取的安全性提升，選擇方案 A

### 方案 C：Cloudflare Tunnel 類中繼服務 — 已否決
- 伺服器對外主動連線到 Cloudflare，不需要開放任何 inbound port，使用者端也不需要安裝軟體
- **否決原因**：流量會經過 Cloudflare 中繼，重新引入「第三方經手資料流量」的疑慮，與當初選擇自架以避開委外處理的初衷矛盾，不予採用

**結論**：若要做遠端/對外存取，目前傾向方案 A（直接開 port + 硬化），但尚未實作。啟動這塊工作前，建議：
1. 確認是否要買正式網域或先用免費 DDNS
2. 重新檢視當時 Windows 主機的限制是否仍然成立（若屆時已換成/新增一台專用的 Linux 主機，上述 Caddy/systemd/ufw 等方案可直接套用；若仍是同一台受限的 Windows 主機，需要重新評估 Windows 上對應的等價工具）
