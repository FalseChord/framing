# 信件模板系統 — 設計文件（第一階段）

## 背景與目標

這是「framing」專案（諮商所行政數位化工具）四個子系統中的第一個，見專案總覽的分解與順序：

1. **信件模板系統**（本文件）
2. 心理師資料庫
3. 個案資料庫
4. 櫃檯值班看板

目前行政人員產信的方式是複製舊信、手動修改關鍵字（日期、姓名等），容易漏改。本階段目標是用網頁工具取代這個流程，同時讓行政人員自己能維護模板內容，不需要工程師介入。

**本階段刻意排除**：真實個案資料尚未進入系統（個案資料庫是第三階段才做的事），本階段只處理信件模板本身與少量心理師名單。

## 範圍

### 涵蓋
- 信件模板的建立、編輯（行政人員自助）
- 依模板 + 使用者填入的欄位，產出完整信件文字
- 一鍵開啟 Gmail 撰寫視窗並帶入收件者/主旨/內文（見「產信」流程）
- 心理師固定名單（下拉選單用，非完整的心理師資料庫功能）
- 操作者身份選擇（辨識目前是誰在操作，對應信件簽名代號；不含密碼驗證，見「部署架構」的說明與取捨）

### 不涵蓋（留給後續階段或另行規劃）
- 個案資料庫、心理師完整資料庫、值班看板
- 信件的 docx/PDF 匯出（目前是頁面顯示文字、人工複製貼上）
- 跨網路存取（見下方「已知限制與後續規劃」）

## 信件種類需求

已知的信件類型，內容差異主要集中在少數段落/詞句，其餘共用：

- **派案類**：媒合信、準備信 — 各自有「一般 / 青壯 / 北捷 / EAP」四種方案變體
- **線上諮詢／通訊諮商預約確認信**
- **諮商時間調整信**
- **團體類**：錄取信、候補信、額滿信

由於同一信件在不同方案間僅有少數段落不同，模板需支援「共用主體 + 條件區塊（依方案變體切換文字）」的設計，不應為每個方案變體各建一份完全獨立的模板。

第一階段成功標準（MVP）：先能產出最常用的幾種信即可，其餘模板可上線後陸續補齊；但「行政人員自助新增/編輯模板」這個能力必須從第一天就存在，不可延後。

## 部署架構（第一階段：本機測試）

**限制條件**：主機是既有的 Windows 電腦（8GB RAM、Intel i5-12400），同時還跑其他文書作業軟體；不能對它做大範圍的系統層級變動；目前只需要在這台主機本機使用，其他行政人員尚不需要透過區域網路連線。

因此第一階段採最簡部署：

- **執行環境**：Node.js（官方 Windows 安裝程式），跑 Next.js（TypeScript）應用程式（前端 + API route 同一個專案）
- **資料庫**：SQLite，單一檔案，隨應用程式一起放在專案資料夾內，不需要另外安裝/啟動資料庫服務
- **網路**：僅監聽 `localhost`，不開放任何對外或對區域網路的 port，不需要調整 Windows 防火牆
- **不處理**：HTTPS、憑證、網域名稱、DDNS — 這些留到之後真的需要跨機器存取時再處理（見下方「已知限制與後續規劃」）
- **操作者身份選擇（無密碼）**：多位行政人員共用這台機器操作時，仍需要知道「目前是誰在操作」，用來對應信件簽名代號與稽核記錄；但考量到使用人數最多 5 人、彼此互相認識、機器不對外開放，**刻意不做密碼驗證**，操作前從固定名單選擇自己即可。取捨：這代表任何人都能選成別人的名字（不論是點錯還是刻意冒用），系統不會阻擋——這是在目前情境下有意識接受的風險，不是疏漏。
- **啟動方式**：測試階段以手動啟動為主（不建置成 Windows 服務常駐），符合「effort 儘量降低」的要求

**規模備註**：個案代號資料量約千筆等級、心理師名單約 30 位以內、使用者最多 5 人——這個規模是後續選擇 SQLite（而非需要獨立服務的資料庫）與省略密碼驗證的主要依據，避免為用不到的規模預先做過度設計。

## 資料模型（SQLite）

- **users**：員工名單。欄位：顯示名稱、簽名代號、建立時間。**不含密碼欄位**——選擇身份不需要驗證，見上方「操作者身份選擇」的取捨說明。
- **therapists**：心理師固定名單。欄位：姓名、是否啟用（供未來心理師資料庫擴充留欄位彈性，本階段先只用姓名做下拉選單）。
- **templates**：信件模板。欄位：類別（媒合信/準備信/預約確認信/…）、方案變體（一般/青壯/北捷/EAP/不適用）、內文（含 `{{placeholder}}` 語法與變體條件區塊）、必填欄位清單（SQLite 無原生陣列型別，以 JSON 字串儲存，應用程式層負責序列化/反序列化）、最後編輯者（關聯 users）、最後編輯時間。
- **letters_log**：稽核記錄。欄位：使用者（關聯 users）、模板（關聯 templates）、產出時間。**刻意不記錄個案代號或信件實際內文**，將留存的個資降到最低。

## 核心流程

### 產信
1. 使用者從固定名單選擇「目前是誰在操作」（無密碼）
2. 選擇信件類型與方案變體
3. 填入欄位：心理師（下拉選單，來自 `therapists`）、日期（日曆選擇器）、個案代號（手動輸入的簡短代稱，非個案全名——這是刻意的隱私設計，不在信件系統內留存可直接識別個案的全名）、收件者 email（手動輸入，**不存入資料庫**，僅用於當次組出 Gmail 連結）、其他該模板定義的欄位
4. 系統依模板 + 填入值渲染出完整信件文字，顯示在頁面上
5. 使用者可以複製文字自行貼上，或點擊「開啟 Gmail 草稿」按鈕——系統組出 `https://mail.google.com/mail/?view=cm&fs=1&to=...&su=...&body=...`（各欄位需 URL 編碼，換行為 `%0A`），瀏覽器開新分頁直接進入 Gmail 撰寫視窗，已帶入收件者/主旨/內文，使用者確認無誤後手動按送出（不會自動寄出）
6. 系統寫入 `letters_log` 一筆稽核記錄（誰、何時、用了哪個模板）——收件者 email 與信件內文不計入此記錄

### 模板編輯
1. 有權限的使用者進入「模板管理」頁面
2. 可新增模板（選擇類別、方案變體、撰寫內文與 placeholder）
3. 可編輯既有模板內文
4. 儲存後立即生效，下次產信直接套用新內容，不需要重新部署程式

## 錯誤處理

- 必填欄位未填 → 擋下產信動作，於畫面上提示缺漏欄位，不允許產出有缺漏的信件
- 模板內文引用了未定義的欄位 → 儲存模板時提示編輯者
- 尚未選擇操作者身份 → 導向身份選擇頁面，無法直接使用模板管理/產信功能

## 測試重點

以目前的規模（~5 位使用者），測試資源優先集中在：

- **模板變數/條件區塊渲染邏輯**：單元測試。這是本系統存在的核心理由（避免漏改日期/姓名），邏輯錯誤的代價最高。
- **必填欄位 JSON 序列化/反序列化邏輯**：單元測試，因為 SQLite 無原生陣列型別，這段轉換若出錯會連帶讓必填檢查失效。
- **端對端驗收**：手動跑一次「選擇操作者 → 選模板 → 填欄位 → 產出信件 → 確認內容正確」流程，作為每次部署前的驗收動作。

## 隱私與資料處理備註

- 本階段刻意不涉及真實個案的完整個資（僅短代稱），降低本階段的資料保護風險。
- **Gmail 草稿功能**：收件者 email 僅在使用者當次操作時手動輸入、用於組成 Gmail 連結，**不寫入資料庫、不計入 `letters_log`**，維持「系統內不留存可識別個案聯絡資訊」的原則。此功能使用 Gmail 網址列參數（`view=cm&to=&su=&body=`）觸發瀏覽器開啟撰寫視窗，資料流向仍是使用者本來就會用的 Gmail 帳號，不是新增的第三方資料處理對象，與先前的雲端/跨境評估無關。這組參數是社群長年沿用、但非 Google 正式文件保證的慣例用法，理論上未來有極小機率失效，屆時退回「複製文字手動貼上」即可，不影響核心功能。
- 本機部署（Windows 主機、不對外開放）是專案更早階段評估雲端方案（Firebase／Vercel／Supabase／GCP）後，因台灣個資法跨境傳輸疑慮與機構同意書未涵蓋委外處理，改為自架的決定 — 完整脈絡與被否決的雲端方案，見同目錄下 `2026-07-16-future-network-architecture-reference.md`。
- 備份：SQLite 是單一檔案，備份即直接複製該檔案（不需要 `pg_dump` 之類的匯出工具）。每晚自動複製一份到本機的備份資料夾，再由人工定期複製到外接硬碟，異地輪換存放（例如帶回家、保險箱），避免單一硬體故障造成資料全失。

## 已知限制與後續規劃

以下是本階段刻意延後、但已經討論過取捨的事項，供後續階段（或本子系統之後的疊代）參考：

- **跨機器/跨網路存取**：目前只有本機使用。若之後需要讓其他行政人員從自己的電腦連線，需要重新評估「開放區域網路 port」vs「其他方案」，並補上 Windows 防火牆設定。
- **對外／遠端存取**：若之後需要遠端存取（例如在家也能用），已討論過三種方案的取捨（直接開 port + 硬化措施 / Tailscale VPN / Cloudflare Tunnel 類中繼服務），最終方向是「直接開 port + 安全硬化」，但尚未實作（HTTPS、網域、防火牆規則、fail2ban 等）。詳見參考文件。
- **正式網域與 HTTPS**：本階段不需要，之後如果要對外開放，需要先決定買網域或用免費 DDNS。
