# 信件模板系統 — 資料補完與功能擴充設計文件（第二階段）

## 背景

第一階段（見 `2026-07-16-letter-templates-design.md`）已完成信件模板系統的基礎架構：SQLite 資料庫、無密碼操作者身份選擇、Handlebars 渲染引擎、模板 CRUD UI、產信 UI 與 Gmail 草稿網址帶入、稽核記錄。心理師與操作者管理 UI 也已在後續工作中完成。

本階段目標：把系統從「架構完成、內容是測試假資料」推進到「可實際上線使用」——補齊信件真實內容、心理師名單欄位，並根據實際內容補齊時發現的落差，擴充模板架構（標題欄位、方案共用主體、多時段、日期自動格式化、粗體/黃底格式化）。

**本階段刻意排除**：UI 美化、頁面導覽/連結整理——另開一輪處理。

## 範圍

### 涵蓋
- Template 資料模型改版：新增標題欄位、改為「每類別一筆、方案差異用條件區塊表示」
- 新的模板內文標記語法：方案條件區塊、多時段條件區塊、粗體＋黃底標記
- 心理師名單新增 email、備註欄位
- 日期/時段選擇器（自動計算星期幾）、多時段輸入（不限筆數）
- 簽名檔自動帶入（含可選的 LINE 聯繫方式勾選）
- 收件者支援多筆、密件副本（BCC）欄位（皆為單次使用，不寫入資料庫）
- 產信頁面改為輸出格式化 HTML（粗體＋淺黃底色），並提供「複製格式化內文＋開啟 Gmail 草稿」一鍵按鈕
- 把已取得的真實信件內容（媒合信、準備信各 6 種方案；線上諮詢預約確認信；通訊諮商預約確認信）匯入系統，取代測試假資料

### 不涵蓋（留給後續階段）
- UI 視覺美化、頁面導覽/連結整理
- 尚未提供內容的類別（諮商時間調整信、錄取信、候補信、額滿信）— 架構就緒，內容之後再補
- Gmail 草稿全自動寫入（IMAP + App Password 方案）— 見「已知限制與後續規劃」

## 內容撰寫原則

使用者提供的範例信（`docs/content/letter-templates-intake.md`）是內容依據，但**不要求逐字 100%還原**。實際整理成模板時，若同一類別底下各方案的內文差異只是措辭上的小不同（例如「進行第一次諮商」vs.「進行第一次伴侶諮商」），且不影響原意，允許適度合併/簡化用字，讓 `[只有]`/`[除外]` 條件區塊盡量單純、不要為了保留每個字的差異而產生大量瑣碎的條件區塊。目標是「共用主體 + 少數關鍵差異」，不是「逐句比對後的精確條件化」。

## 資料模型變更

### Template（改版）

從「每個類別＋方案組合各一筆」改為「每個類別一筆，方案差異寫在內文的條件區塊裡」：

- `category`：類別（媒合信、準備信、線上諮詢預約確認信、通訊諮商預約確認信…）
- `subject`：標題（主旨），語法與 `body` 相同，支援 `{{placeholder}}` 與條件區塊
- `body`：內文，共用文字＋條件區塊（語法見下方）
- `variants`：JSON 字串陣列，這個類別支援哪些方案，例如 `["一般","伴侶","青壯","重大災害","EAP","公益"]`；沒有方案差異的類別存 `["不適用"]`
- `requiredFields`：JSON 字串陣列（沿用第一階段設計，行政人員在模板編輯時自行宣告必填欄位）
- `updatedAt`、`updatedById`、`updatedBy`：沿用

**移除**：原本的 `variant`（單一字串）欄位，改為上述 `variants`（陣列）。

**已知真實方案清單**（與第一階段設計文件推測的「一般/青壯/北捷/EAP」不同，以實際貼上的內容為準）：
媒合信、準備信 → 一般、伴侶、青壯、重大災害、EAP、公益（無「北捷」方案）。

### Therapist（新增欄位）

- `email`（可選 String）：管理聯繫用，不參與信件渲染
- `note`（可選 String）：自由文字備註，無固定分類

### 不新增資料表的部分

- **簽名檔**：固定文字＋`{代號}`（對應 `session.signature`），渲染時直接附加在信件內文最後，不存資料庫、不算進模板內文本身。
- **收件者／密件副本（BCC）**：兩者皆可新增多筆，且皆為選填——允許「只填 BCC、不填收件者」的情況（例如只需要副本通知內部人員，尚未確認個案 email 的狀況）。跟現行單一收件者欄位一樣，僅在產信當次於畫面輸入，用於組出 Gmail 網址參數，**絕不寫入資料庫或稽核記錄**。

## 模板內文標記語法（新設計）

延續第一階段「行政人員自助編輯、不需要寫程式」的原則，內文用一套簡單的方括號標記語法，而非直接寫 Handlebars 條件語法。系統內部仍用 Handlebars 渲染變數代換，但方案/時段條件會先經過一層轉換器（preprocessor）處理成 Handlebars 再渲染。

### 1. 變數代換（沿用第一階段）
`{{caseRef}}`、`{{therapistName}}`、`{{sessionDate}}` 等，不變。

### 2. 方案條件區塊（新增）
```
[只有 EAP]
本次服務由貴公司 EAP 方案支付費用。
[/只有]

[除外 EAP]
期待與您見面。
[/除外]
```
- `[只有 A、B]...[/只有]`：只在選擇的方案是 A 或 B 時顯示
- `[除外 A、B]...[/除外]`：除了 A、B 以外的方案都顯示
- 多個方案用「、」分隔
- 區塊外的文字視為共用內容，任何方案都會顯示

### 3. 多時段條件區塊（新增）
用於「候選時段只有一個」跟「兩個以上」需要不同措辭的情境（如媒合信-青壯方案範例）：
```
[單一時段]
第一次諮商時間預計安排在 {{sessionSlots}}
請問您是否方便前來？再麻煩您撥冗回覆。
[/單一時段]

[多個時段]
需要和您確認第一次諮商時間，想詢問您以下時間是否方便，
{{sessionSlots}}
[/多個時段]
```
- 是否顯示由該模板的時段欄位實際填入筆數決定（1 筆 vs. 2 筆以上），不需要行政人員手動判斷
- `{{sessionSlots}}` 這個特殊變數會依筆數自動格式化：1 筆時輸出單一時段字串；2 筆以上時自動輸出項目符號列表（`◉ {時段1}` 換行 `◉ {時段2}` ...），行政人員不需要自己排版符號列表

### 4. 粗體＋淺黃底色標記（新增）
```
已為您媒合心理師 **{{therapistName}}**，
```
- `**文字**`（雙星號包住，借用大家熟悉的 Markdown 粗體語法）＝這段文字用粗體＋淺黃底色顯示
- 可以包住固定文字（如「收到信件煩請回信告知」）或包住變數（如 `**{{therapistName}}**`），兩種混用都支援
- 本系統目前的房規是「粗體」跟「淺黃底色」永遠一起出現，所以用一組語法同時代表兩者，不分開標記

### 日期／時段欄位（新增欄位型別）

必填欄位如果宣告為「日期時段」型別，模板編輯與產信頁面會改用專用元件而非純文字輸入：
- 選擇日期＋起訖時間，系統自動算出星期幾並組成字串代入 `{{欄位名}}`
- **輸出不含年份**（例如 `7/22 (三) 19:00-20:20`），與貼上的範例信格式一致；年份日後若有需要再另行評估
- 「多時段」型別欄位（用於 `{{sessionSlots}}`）：可自由新增/刪除任意筆日期時段組合，沒有固定上限，每筆格式規則與單一日期時段欄位相同

## 產信流程變更

1. 選擇操作者、類別、方案（若該類別有多個方案，顯示方案下拉選單；只有一種內容則不顯示）
2. 依模板宣告的必填欄位動態產生表單，日期/時段類欄位使用專用選擇器
3. 填入收件者與密件副本（皆可新增多筆、皆為選填，允許只填密件副本的情況）、勾選是否附加官方 LINE 聯繫方式那一行
4. 系統渲染出格式化 HTML 預覽（粗體/淺黃底色已套用），畫面同時顯示純文字版本供核對
5. 點擊「複製格式化內文並開啟 Gmail 草稿」：
   - 將格式化 HTML（連同純文字備援格式）寫入剪貼簿
   - 同時開新分頁開啟 Gmail 撰寫視窗，網址參數帶入收件者（多筆以逗號分隔）、密件副本、主旨（純文字，無格式）
   - 內文請操作者手動貼上（Ctrl+V）——因為 Gmail 的撰寫網址參數只支援純文字，無法透過網址帶入粗體/底色，這是 Gmail 本身的限制，貼上是目前唯一能保留格式的方式（見下方「已知限制與後續規劃」的 Gmail API／IMAP 方案評估）
6. 系統寫入 `letters_log` 稽核記錄，同前一階段設計，不含收件者/內文/個案代號

## 錯誤處理（新增部分）

- 內文引用了不存在的方案名稱（例如打錯字）於 `[只有]`/`[除外]` 區塊 → 儲存模板時提示編輯者
- `[單一時段]`/`[多個時段]` 區塊只能搭配宣告為「多時段」型別的欄位使用；模板宣告了這類區塊但沒有對應欄位 → 儲存時提示編輯者
- `**...**` 標記未正確配對（缺少結尾）→ 儲存時提示編輯者，避免渲染出跑版內容

## 測試重點（延續第一階段原則，新增部分）

- **方案條件區塊解析**：單元測試，涵蓋單一方案、多方案、除外邏輯、同一模板中多次出現的情況
- **多時段條件區塊與 `{{sessionSlots}}` 自動格式化**：單元測試，涵蓋 0/1/2/多筆情境
- **星期幾自動計算**：單元測試，涵蓋跨年、跨月邊界日期
- **`**文字**` 轉換成粗體＋底色 HTML**：單元測試，涵蓋巢狀變數代換後的文字
- **剪貼簿複製（HTML + 純文字雙格式）**：端對端手動驗收，實際貼到 Gmail 撰寫視窗確認格式保留

## 已知限制與後續規劃

延續第一階段文件的既有項目，新增：

- **Gmail 草稿全自動寫入（含格式）**：已評估四種方案——
  1. Gmail API + OAuth：需要每位操作者個別授權、測試模式權杖 7 天過期、正式上線需 Google 審核（含網域與隱私權政策），與目前「本機、無網域」的部署決定衝突，不採用。
  2. 瀏覽器擴充套件自動注入：不需要密碼，但要另外安裝/維護擴充套件，且依賴 Gmail 網頁結構，Google 改版可能失效。
  3. Playwright 本機瀏覽器自動化：不需要存密碼、不需要另裝擴充套件，但同樣依賴 Gmail 網頁結構、且可能被 Google 判定為可疑登入行為而卡住。
  4. **IMAP + App Password（目前最可能的未來升級路徑）**：共用信箱帳號只要能開啟兩步驟驗證，就能產生一組應用程式密碼，讓系統透過 IMAP 直接把格式化好的信件寫入該帳號的草稿匣，達到真正一鍵完成（無需複製貼上）。密碼**不存資料庫**，只存在主機的 `.env`（不進備份、不進 git），風險範圍等同於「有實體/系統存取權的人可操作這台主機」，與目前手動操作的風險等級相當。**目前卡住的前提**：共用帳號尚未開啟兩步驟驗證，且行政人員可能沒有權限自行開啟，需要向管理者確認。若之後確認可行，可作為本階段方案 A（複製貼上）的直接升級，不需要重新設計資料流程。
  - **本階段決定**：先採用方案 A（複製格式化內文＋手動貼上），不依賴前述任何一種自動化方案。
- **尚未取得真實內容的類別**：諮商時間調整信、錄取信、候補信、額滿信——架構（標題欄位、方案條件區塊等）已支援，待內容補齊後即可比照媒合信/準備信的方式匯入，不需要額外開發。
