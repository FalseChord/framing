# 設計文件：全站視覺設計方向

**日期**：2026-07-19
**狀態**：待使用者審閱
**關聯文件**：[2026-07-19-independent-template-variants-design.md](./2026-07-19-independent-template-variants-design.md)

## 背景

前一份 spec（範本變體獨立儲存）明確把視覺設計排除在範圍外。使用者後續提出希望一併重新設計**全站**（不只是範本管理／產生信件頁）的視覺呈現，於是用 `ecc:frontend-design-direction` 這個 skill 訂出方向，補齊前一份 spec 刻意留白的部分。

本文件涵蓋的頁面：`/`（首頁）、`/select`（身份選擇）、`/generate`（產生信件）、`/templates`（模板管理）、`/therapists`（心理師名單管理）、`/users`（操作者名單管理）。

## 設計方向

- **Purpose（用途）**：讓行政人員快速、正確地管理範本與名單、產生信件寄出，重點是「不出錯」，不是展示型或行銷型介面。
- **Audience（使用者）**：固定 ≤5 位行政人員，每天/每週重複操作，需要一眼掃到「現在在哪個類別、哪個變體」「這欄是不是必填」「現在是誰在操作」。
- **Tone（調性）**：務實密集型（utilitarian, dense, quiet, scannable）——安靜、資訊密度高但分區清楚，不用行銷風格的大標題、漸層、裝飾性動畫。
- **Memorable detail（辨識度細節）**：借用信件內文既有的 `**淺黃底粗體**` 標記樣式，作為整套介面的強調色語言，讓「工具裡看到的顏色」跟「操作者實際產出的信件裡看到的顏色」互相呼應。
- **Constraints（限制）**：專案目前沒有安裝任何 CSS 框架或 UI 元件庫，本次設計刻意不新增任何前端依賴，全部用 Next.js 內建能力（全域 CSS、CSS Modules、原生 HTML 元素）實作，延續專案一貫「單機小規模、避免過度設計」的原則。

## 色彩

以 CSS 自訂屬性（CSS custom properties）定義於全域樣式表，供全站共用：

| 用途 | 色彩方向 | 說明 |
|---|---|---|
| 背景 `--color-bg` | 暖灰白（約 `#fafaf9`） | 避免死白，與琥珀強調色搭配更協調 |
| 卡片/面板 `--color-surface` | 純白 `#ffffff` | 用於表單、清單容器 |
| 邊框 `--color-border` | 柔和暖灰 `#e5e2dc` | |
| 主文字 `--color-text` | 近黑暖色 `#1f1d1a` | |
| 次要文字 `--color-text-muted` | 中灰 `#6b6459` | 用於備註、輔助說明文字 |
| 強調背景 `--color-accent-bg` | 淺黃 `#fde68a` 類 | 用於「目前選中」項目背景、focus 外框，呼應信件內的粗體反白樣式 |
| 強調主色 `--color-accent-strong` | 深琥珀 `#b45309` 類 | 用於主要按鈕背景／連結色，確保與白色文字有足夠對比度 |
| 危險色 `--color-danger` | 柔和紅 | 專用於「停用」「刪除」等危險操作，與強調色明確區隔，避免整頁被單一色相（琥珀）蓋滿 |

色盤刻意維持多層次（中性為主、琥珀為強調、紅色為警示），符合「避免整頁被單一色相支配」的設計原則。

## 排版與間距

- 全站使用系統字體堆疊（不載入額外字型檔）：`-apple-system, BlinkMacSystemFont, "Segoe UI", "PingFang TC", "Microsoft JhengHei", sans-serif`
- 範本內文編輯框使用等寬字體（沿用前一份 spec 的決定：`ui-monospace, SFMono-Regular, Menlo, monospace`）
- 全站內容寬度統一為 960px 置中（取代原本各頁各自決定或完全無限制的狀況）
- 間距採 8px 基準的等比例尺（8 / 12 / 16 / 24 / 32px），維持「密集但不擁擠」的密度

## 導覽列（新增的站台結構）

- 目前完全沒有跨頁導覽——只有首頁 `/` 有一份連結列表，進到其他頁面後只能靠瀏覽器上一頁切換，這是本次一併修正的結構性缺口。
- 新增一個固定頂部導覽列，出現在所有**已選定身份**的頁面：左側系統名稱＋「產生信件／模板管理／心理師管理／操作者管理」連結，右側顯示目前操作者（姓名＋簽名代號），讓操作者隨時知道「現在是我在操作」——這點在無密碼登入的系統裡格外重要。
- `/select`（身份選擇頁）不顯示導覽列，因為那個當下還沒有身份可顯示。
- 技術實作：用 Next.js App Router 的路由群組（route group，例如 `(app)/`）把需要導覽列的頁面包進共用 layout，`/select/page.tsx` 留在群組外、不受影響；不新增任何路由或狀態管理套件。

## 元件與互動模式（刻意不新增依賴）

- 模板管理頁「依類別分組、展開查看變體」：使用瀏覽器原生 `<details>/<summary>` 元素實作，天生支援鍵盤操作與螢幕閱讀器，不需要額外的 accordion 套件或自訂 JS 狀態。
- 各管理頁面（範本／心理師／操作者）的清單維持現有的 `<ul><li>` 結構，加上 hover 狀態與更清楚的「編輯」按鈕樣式，不改成複雜的資料表格元件。
- 按鈕分三種語意樣式，用共用 CSS class 定義（非 CSS-in-JS）：
  - 主要（`--color-accent-strong` 底、白字）：送出、儲存、產生信件等主要動作
  - 次要（外框、中性色）：取消等次要動作
  - 危險（外框、`--color-danger`）：停用等破壞性動作
- 表單維持現有的「label 在上、輸入框在下」直式排列，不引入分欄式或精靈式（wizard）表單。

## 動效

- 僅在 hover／focus／選中狀態使用約 150ms 的背景色過渡，不做頁面轉場、進場動畫或任何裝飾性動效——內部工具的操作效率優先於視覺效果。

## 無障礙與響應式

- 沿用現有的 `<label>` 包裹 `<input>` 慣例（已符合基本語意關聯）；focus 外框使用強調色，確保鍵盤操作時清楚可見。
- 文字對比度：深琥珀（強調主色）用於白底上的文字/按鈕，需確保達到基本可讀對比；淺黃僅用於背景色塊，搭配深色文字，不單獨用淺黃當文字色。
- 響應式僅需基本保證（單機 Windows 桌機部署，非行動裝置為主要使用情境）：導覽列連結允許換行、表單維持單欄堆疊即可，不特別做手機版斷點或漢堡選單。

## 實作範圍（供後續 writing-plans 參考）

- 新增 `src/app/globals.css`：定義上述 CSS 自訂屬性、基礎 reset、共用按鈕/表單 class，於根 `layout.tsx` 引入一次。
- 新增共用導覽列元件（例如 `src/components/AppNav.tsx`），讀取目前 session 顯示操作者姓名與簽名代號。
- 重新調整路由結構為路由群組，把 `/generate`、`/templates`、`/therapists`、`/users`、`/` 納入共用 layout；`/select` 維持獨立。
- 套用上述樣式規範到既有 5 個頁面的既有標記，以及前一份 spec 中規劃的範本管理頁新結構（類別分組展開、放大的內文編輯框）。
- 不涉及後端邏輯、資料模型、API 行為變更——與前一份 spec（範本變體獨立儲存）是各自獨立、互不衝突的改動範圍，兩份 spec 可以合併成一份實作計畫，或分兩個階段實作。

## 已知取捨

- 用原生 `<details>/<summary>` 而非自訂 accordion 元件，換來的是無障礙與零依賴，但樣式客製化彈性較低（例如展開/收合動畫效果有限）；對務實密集型調性來說是合理取捨。
- 導覽列需要知道目前操作者身份，這代表共用 layout 需要能存取 session 資料（server component 讀取 `getSession()`）；技術上可行，但要留意這是本次新增的一個「跨頁共用狀態讀取」，屬於本次設計方向帶來的新結構，非既有模式的延伸。
